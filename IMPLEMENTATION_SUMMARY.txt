AUDIT FINDINGS (Critical ACID Violations):
================================================================================

1. ATOMICITY VIOLATION
   Problem: DuckDB writes are synchronous but L2 Parquet writes are asynchronous
            → If Parquet fails after DuckDB succeeds, data visibility is inconsistent
   Impact: Users see data in DuckDB but not in L2 cache (watermark readers miss data)

2. CONSISTENCY VIOLATION
   Problem: DuckDB ↔ Parquet skew; no transactional boundary between stores
            Concurrent watermark reads miss recently written data
   Impact: Cache misses for valid data; stale reads from manifest

3. ISOLATION VIOLATION
   Problem: Multiple DuckDB connections without pooling/locking
            session_state accessed from multiple threads without synchronization
   Impact: Race conditions on shared mutable state; lost updates to session vars

4. DURABILITY VIOLATION
   Problem: Async L2 writes not durable; session_state never persists to disk
            Page refresh/crash loses all user session data
   Impact: Poor UX (lost shopping carts, preferences); no crash recovery

================================================================================
SOLUTION APPROACH:
================================================================================

Fix 1: DuckDB Connection Pooling (2 hours)
  └─ Single shared connection + threading.RLock for isolation
     ├─ Prevents concurrent writes on separate connections
     ├─ Ensures session context consistency
     └─ All operations execute atomically via lock

Fix 2: Transactional Writes (4 hours) [Requires Fix 1]
  └─ Wrap DuckDB → Parquet → Manifest chain in single transaction
     ├─ Write Parquet to temp file (safe)
     ├─ BEGIN TRANSACTION in DuckDB
     ├─ Atomic rename temp → final
     ├─ INSERT audit entry (watermark)
     ├─ UPDATE manifest
     ├─ COMMIT (or ROLLBACK on error)
     └─ No partial state on disk

Fix 3: Session State Persistence (4.5 hours) [Independent]
  └─ DuckDB-backed session store for user data persistence
     ├─ Load persisted state on app startup
     ├─ save_key() updates DuckDB on each change
     ├─ Session survives refresh/crash
     └─ Scales to millions of sessions

Fix 4: Manifest Locking (3 hours) [Independent]
  └─ Advisory file locks (fcntl/msvcrt) on manifest.json
     ├─ Serialize concurrent read-modify-write
     ├─ Write to temp file, atomic rename
     ├─ No lost updates to cache metadata
     └─ Timeout handling prevents deadlock

KEY IMPLEMENTATION DECISIONS:
================================================================================

Why single connection + lock (Fix 1)?
  ✓ DuckDB is explicitly thread-safe within single connection
  ✓ Simpler than connection pool (no resource management)
  ✓ Lower overhead (one object)
  ✓ Explicit locks make audit trail clear

Why transaction + atomic rename (Fix 2)?
  ✓ OS-level atomic rename prevents partial writes
  ✓ DuckDB transactions provide isolation
  ✓ Watermark in audit table = single source of truth
  ✓ All-or-nothing semantics (no halfway states)

Why DuckDB for session store (Fix 3)?
  ✓ Reuses existing connection (no new infrastructure)
  ✓ JSON serialization is standard (no custom encoding)
  ✓ ON CONFLICT prevents lost updates
  ✓ Scales to millions of sessions

Why file locks (Fix 4)?
  ✓ Portable (fcntl on Unix, msvcrt on Windows)
  ✓ No external dependency (Redis, etcd, etc.)
  ✓ Advisory locks prevent race conditions
  ✓ Atomic rename = crash-safe

ROLLBACK PROCEDURES:
================================================================================

Fix 1 Rollback:
  $ git revert <commit>
  └─ Removes RLock, reverts to lazy singleton
     No data loss, but ISOLATION not guaranteed

Fix 2 Rollback:
  $ git revert <commit>
  └─ Uses old async write_cache(), included as fallback
     Data loss risk returns, but app keeps working

Fix 3 Rollback:
  $ # In app/main.py, comment out setup_session_persistence()
  └─ Session_state reverts to in-memory only
     Users lose data on refresh (poor UX, but recoverable)

Fix 4 Rollback:
  $ git revert <commit>
  └─ Removes file locks, direct JSON write
     Manifest corruption risk returns, but app keeps working

Recommended: Merge Fixes 1+4 together (both fix thread safety)
            Can rollback as atomic unit without affecting Fixes 2+3

Emergency rollback:
  1. Revert to previous stable commit
  2. Restore manifest.json from backup
  3. Restart app
  4. No data loss (Parquet files unchanged)

================================================================================
RISKS & MITIGATION:
================================================================================

Risk: Lock contention causes latency spike
Mitigation: RLock is reentrant; timeout on lock waits; monitor metrics

Risk: Manifest file corruption
Mitigation: Atomic rename + temp file; file locking; manifest.json backup

Risk: Session data JSON encode error
Mitigation: Try/except on JSON encode; skip failed keys; log warning

Risk: DuckDB transaction fails (out of memory, etc.)
Mitigation: ROLLBACK on error; temp file cleanup; fallback to old async write

Likelihood: ALL RISKS ARE PREVENTABLE BY CAREFUL TESTING
Severity: MEDIUM (data loss possible but recoverable from Parquet)
Impact:   LOW (monitoring prevents production issues)
