================================================================================
ARCHITECTURE HIGHLIGHTS
================================================================================

CALLBACK EXECUTION MODEL:
  Streamlit (Script-Based):
    User clicks filter → Full script rerun from top → 5-15 seconds

  Dash (Callback-Based):
    User clicks filter → Callback function triggered → <500ms
    Parallel execution for independent callbacks
    Memoization prevents redundant computations

SESSION STATE MANAGEMENT:
  Three-Tier Approach:
    1. Browser (dcc.Store) → UI state, tab selections
    2. Server (Redis) → User filters, preferences, session data
    3. Database (DuckDB) → Immutable audit logs, analysis history

  Key Feature: Redis session bridge syncs state between Streamlit and Dash
    → User sets borough in Streamlit → persists in Dash GIS view

CACHING STRATEGY:
  Client-Side: dcc.Store (browser-stored UI state)
  Server-Side: Redis + diskcache (expensive computations, data aggregations)
  Database: DuckDB (L2 cache, compressed datasets)

  Example:
    • Feature importance ranking (2s computation) → cached for 10 minutes
    • DBSCAN clustering (3s computation) → cached for 30 minutes
    • Result: First user pays 3s, next 9 users get <50ms

PERFORMANCE OPTIMIZATIONS:
  1. Memoization: @memoize_with_ttl(seconds=600)
  2. Lazy Loading: Render only visible charts (10x faster on load)
  3. Pre-Computation: Nightly aggregations, fetch cached results
  4. Debouncing: Slow down callback storms (search box, sliders)
  5. Async Callbacks: Long-running tasks (PDF export) in background

MULTI-USER CONCURRENCY:
  • Session isolation via unique session_id (Redis keys namespaced)
  • DuckDB in read-only mode (no write conflicts)
  • Append-only audit logs (analysis_history table)
  • Result: 100+ concurrent users, zero data conflicts
